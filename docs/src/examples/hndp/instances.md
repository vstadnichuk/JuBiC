# HNDP Instances

The HNDP example uses generated instance families to turn the mathematical model
from the motivation page into concrete computational test cases.

## Instance Types

The currently documented variants are:

- `constrained_shortest_path`: a single-layer graph where all arcs are
  first-level decision arcs.
- `single_layer_k_decision_only`: a single-layer alternative where only ``k``
  sampled arcs are first-level decision arcs; routing costs are not scaled.
- `layered_*` competition instances: two-layer graphs with a competitor layer
  and an operator-controlled layer.
- `single_layer_k_competition`: a single-layer graph where ``k`` sampled arcs
  are decision arcs and also receive the competition cost scaling.

Each variant specifies how the abstract objects in the formulation are filled
with concrete data:

- the graph ``G = (V,E)``,
- the decision set ``A \subseteq E``,
- the user origins and destinations ``(o_u,d_u)``,
- follower costs ``c^u_a``,
- operator-side coefficients ``r^u_a``,
- optional weights ``w^u_a`` and bounds ``W_u``,
- construction costs ``p_a`` and optional construction budgets.

The implementation of these variants is in
`examples/HNDP/hndp_network_generation.jl`. The generated application instances
are then translated into JuBiC solver models by
`examples/HNDP/hndp_model_generation.jl`.

## Constrained Shortest Paths

In the `constrained_shortest_path` family, ``G`` is one base graph topology.
The current generator supports this family for the Sioux Falls topology.

The generated data is:

- ``A = E``: all arcs of the base graph are first-level decision arcs.
- ``(o_u,d_u)``: sampled feasible origin-destination pairs.
- ``c^u_a``: random nonnegative follower costs.
- ``r^u_a``: random operator-side coefficients.
- ``w^u_a``: random nonnegative arc weights if `length_constrained = true`.
- ``p_a``: construction costs sampled from `construction_cost_min` to
  `construction_cost_max`.

User data can be generated in two modes. With `shared`, all users share the
same cost, risk, and weight matrices. With `per_user`, each user receives its
own generated arc data.

If length constraints are enabled, the mathematical parameter ``\alpha`` is
written as `alpha` in configuration files and benchmark output. For a user with
shortest possible weight ``W_u^{min}`` and a family-specific upper reference
value ``W_u^{max}``, the generated bound is

```math
W_u =
\left\lceil
W_u^{min} + \alpha (W_u^{max} - W_u^{min})
\right\rceil .
```

Thus, ``\alpha = 0`` gives the tightest generated bound and ``\alpha = 1`` gives the
loosest generated bound used by the generator. If `length_constrained = false`,
then ``w^u`` and ``W_u`` are not used.

The optional `availability_budget_fraction` is a fraction of the decision arcs
``A`` that may be selected. It is converted to an integer budget by

```math
B = \left\lfloor \texttt{availability\_budget\_fraction} \cdot |A| \right\rfloor .
```

If it is omitted, the fraction is treated as `1.0`, so the budget is ``|A|``.
Alternatively, an integer `availability_budget_count` can be specified directly.

## Single-Layer `k` Decision-Only

`single_layer_k_decision_only` is the single-layer alternative to
`constrained_shortest_path` where only ``k`` arcs are controllable instead of the
whole graph. The mode is selected by
`single_layer_arc_mode = "decision_only"`. The base graph is again used as
``G``.

The generated data is:

- ``A``: ``k`` sampled arcs from ``E``. If ``k > |E|``, all arcs are used.
- ``(o_u,d_u)``: sampled feasible OD pairs on the base graph.
- ``c^u_a``: random costs, without ``\beta`` scaling.
- ``r^u_a``: positive risk coefficients on all arcs.
- ``w^u_a`` and ``W_u``: generated by the same ``\alpha`` rule when length
  constraints are enabled.
- ``p_a``: construction costs on sampled decision arcs.

Here ``k`` only controls first-level controllability. It does not change the
cost parameters seen by users. The generator supports different ``\alpha``
values for this mode in the same way as for `constrained_shortest_path`, as long
as `length_constrained = true`.

## Layered Competition

For topology names starting with `layered_`, ``G`` is built from a base graph by
creating two copies of it:

- a competitor layer,
- an operator layer,
- and transfer arcs between corresponding nodes.

The generated data is:

- ``A``: all arcs inside the operator layer.
- ``(o_u,d_u)``: sampled feasible OD pairs, placed on the competitor layer.
- ``c^u_a``: random costs, modified by the layer rules below.
- ``r^u_a``: random operator-side coefficients, kept only where the operator
  receives a contribution.
- ``w^u_a`` and ``W_u``: generated by the same ``\alpha`` rule as above when
  length constraints are enabled.
- ``p_a``: construction costs on operator-layer arcs.

The mathematical parameter ``\beta`` is written as `beta` in newer
configuration files and as `competitor_cost_factor` in older ones. For
operator-layer decision arcs, costs are scaled by

```math
c^u_a \leftarrow \operatorname{round}(\beta c^u_a)
\qquad a \in A .
```

Values ``\beta < 1`` make the operator-controlled layer cheaper for users.
Values near ``1`` remove most of that artificial cost advantage.

Transfer arcs between the two layers have zero cost and zero weight in the
directions used by the generator. Non-operator arcs have zero operator-side
coefficient. On operator-layer arcs, negative values of ``r^u_a`` represent
profit earned by the operator when users choose those arcs.

The optional `availability_budget_fraction` is converted into an integer budget
on the operator-layer decision arcs using the same floor rule described above.

## Single-Layer `k` Competition

For non-layered competition topology names, the generator keeps the original
base graph as ``G``. The mode `single_layer_k_competition` is selected by
`single_layer_arc_mode = "competition"`.

The generated data is:

- ``A``: ``k`` sampled arcs from ``E``. The requested value is provided by
  `decision_arc_count`; if ``k > |E|``, all arcs are used.
- ``(o_u,d_u)``: sampled feasible OD pairs on the base graph.
- ``c^u_a``: random costs, with ``c^u_a`` scaled by ``\beta`` for ``a \in A``.
- ``r^u_a``: nonzero only on sampled decision arcs.
- ``w^u_a`` and ``W_u``: generated by the same ``\alpha`` rule when length
  constraints are enabled.
- ``p_a``: construction costs on sampled decision arcs.

Thus ``k`` controls both the number of arcs in the decision set and the number
of arcs whose user costs are modified by ``\beta``. In this single-layer
competition mode, operator-side coefficients are nonzero only on sampled
decision arcs. The generator supports different ``\alpha`` values for this mode
as well when `length_constrained = true`; if `length_constrained = false`,
different `alpha` entries are ignored because no weight bounds are generated.

## Network-Generation Configs

HNDP experiments can be generated from JSON configuration files. The network
generation config defines the instance variants and parameter grids. The
pipeline reads this JSON file, expands list-valued fields into a Cartesian
product, and generates one concrete HNDP instance for each combination.

Solver-specific models are not built at this stage. The generated network data
is passed to the model-generation step afterwards, where the selected JuBiC
formulation is constructed.

A minimal layered competition network-generation configuration has the shape:

```json
{
  "parameter_seeds": [42],
  "instances": [
    {
      "name": "layered_demo",
      "instance_type": "competition",
      "topologies": ["layered_sioux_falls"],
      "nusers": [100],
      "length_constrained": [true],
      "alpha": [1.0],
      "beta": [1.0, 0.8],
      "construction_cost_min": 0,
      "construction_cost_max": 0,
      "availability_budget_fraction": [0.5]
    }
  ]
}
```

The full set of supported generation fields is documented in
`examples/HNDP/NETWORK_GENERATION.md`.
