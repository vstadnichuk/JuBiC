# Benders-like Cuts with Lagrangian Coefficients (`BlCLag`)

`BlCLag` uses the same high-point-relaxation structure as `BlC`, but JuBiC
generates the big-M-style cut coefficients automatically from auxiliary
connector problems. The user therefore does not pass a `big_m(a, sub_name)`
callback to the master wrapper.

## Mathematical Structure

`BlCLag` starts from the high-point relaxation (HPR) of the full bilevel
problem. The HPR contains the upper-level constraints together with the
feasibility constraints of all follower problems, while follower optimality is
enforced later through lazy cuts. For JuBiC's multi-follower setting, the HPR
therefore contains the first-level variables `x` and copied follower variables
`\hat y^k` for all followers `k`.

The relaxation has the structure

```math
\begin{aligned}
\min_{x,\hat y}\quad
    & F(x) + \sum_{k \in \mathcal{K}} r_k(\hat y^k) \\
\text{s.t.}\quad
    & x \in X, \\
    & \hat y^k \in Y_k, && k \in \mathcal{K}, \\
    & \hat y^k_a \le x_a, && a \in A,\; k \in \mathcal{K}.
\end{aligned}
```

`BlCLag` restores follower optimality lazily. For each follower, JuBiC solves a
connector problem that computes coefficients for a cut of the form

```math
f_k(\hat y^k) \le s_k + \sum_{a \in A} k_{ka}(1 - x_a).
```

The difference to `BlC` is that `s_k` and `k_{ka}` are generated by JuBiC rather
than supplied through a user-defined big-M callback.

## JuBiC Object Structure

`solve_instance!(inst, params::BlCLagparam)` expects:

- `inst.master isa BlCLagMaster`
- `inst.subproblems` to be a collection of bilevel-capable subsolver objects

The master model passed to `BlCLagMaster` is the HPR model. Unlike
`BlCMaster`, it does not take a `big_m(a, sub_name)` callback.

## Compatible Subsolvers

`BlCLag` requires subsolvers that can solve the bilevel separation routines used
inside the connector problem. Plain follower-only subsolvers are not sufficient.

The currently implemented supported subsolvers are:

- `SubSolverBlCJuMP`
- `SubSolverMiBS`

## Solver Parameters

`BlCLag` is configured with `BlCLagparam`. The most relevant inputs are:

- `solver`: the MIP solver wrapper used for the HPR master and connector models.
- `debbug_out`: whether instance-level debug artifacts should be written.
- `output_folder_path`: output directory used when output logs are enabled.
- `file_format_output`: file format used for exported JuMP models, for example `"lp"` or `"mps"`.
- `pareto`: Pareto-cut mode used inside connector-based cut generation.
- `warmstart`: whether connector state is reused across callback solves.
- `runtime`: runtime limit in seconds.
- `parallel_separation`: whether connector separations are solved in parallel for one incumbent master solution.
- `threads_master` and `threads_sub_con`: thread limits for the master MIP and connector/subsolver-side work.

Unlike `BlC`, `BlCLag` does not take user-provided big-M coefficients in the
master wrapper.

If `parallel_separation = true`, JuBiC requires `threads_sub_con = 1`. In that
mode the parallelism comes from solving several connector/subsolver tasks
concurrently, so each worker-side model must remain single-threaded.

## Minimal Working Example

The example solves the same small bilevel model as the `BlC` page, but the
master wrapper does not receive user-provided big-M coefficients.

```julia
using JuBiC
using JuMP

function build_blclag_example()
    solver = GurobiSolver()
    optimizer = () -> get_next_optimizer(solver)

    A = [1, 2]
    sub_name = "Sub0"

    hpr = Model(optimizer)
    @variable(hpr, x[A], Bin)
    @variable(hpr, yh[A], Bin)
    @constraint(hpr, yh[1] <= x[1])
    @constraint(hpr, yh[2] <= x[2])
    @constraint(hpr, yh[1] == 1)
    @objective(hpr, Min, x[1] - x[2] + 10 * yh[2])
    second_level_obj = @expression(hpr, -yh[1] - yh[2])

    master = BlCLagMaster(
        hpr,
        A,
        Dict(a => x[a] for a in A),
        [sub_name],
        Dict(sub_name => second_level_obj),
    )

    sub_model = Model(optimizer)
    set_silent(sub_model)
    @variable(sub_model, y[1:2], Bin)
    @constraint(sub_model, y[1] == 1)
    follower_obj = @expression(sub_model, -y[1] - y[2])
    @objective(sub_model, Min, follower_obj)
    master_obj_term = @expression(sub_model, 10 * y[2])

    subsolver = SubSolverBlCJuMP(
    sub_name,
    sub_model,
    A,
    y,
    master_obj_term,
    follower_obj,
    solver,
    a -> 100.0,
)

    return Instance(master, [subsolver]), solver
end

instance, solver = build_blclag_example()
params = BlCLagparam(
    solver,
    false,
    mktempdir(),
    "lp",
    PARETO_OPTIMALITY_ONLY,
    true,
    600.0,
)

stats = solve_instance!(instance, params)
println(stats.data["Opt"])
```

The coefficient callback in `SubSolverBlCJuMP` belongs to that subsolver's
internal bilevel oracle. It is not a `BlCLagMaster` big-M callback.
